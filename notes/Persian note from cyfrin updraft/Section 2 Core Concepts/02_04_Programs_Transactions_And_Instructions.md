# Understanding Solana Programs: The Executable Stateless Account (درک برنامه‌های سولانا: حساب اجرایی بدون وضعیت)

برای تسلط بر توسعه در **Solana (سولانا)**، ابتدا باید معماری خاص برنامه‌های آن را درک کرد. برخلاف **Smart Contract (قرارداد هوشمند)** در بلاک‌چین‌های مبتنی بر **EVM (ماشین مجازی اتریوم)**، برنامه‌های سولانا با **تفکیک سخت‌گیرانه بین منطق و داده** طراحی شده‌اند.

در سطح بنیادی، یک **Program (برنامه)** در سولانا به شکل زیر تعریف می‌شود:

**Executable Stateless Account (حساب اجرایی بدون وضعیت)**

---

## Executable (قابل اجرا)

در سولانا، مانند هر چیز دیگری، یک **Program (برنامه)** نیز داخل یک **Account (حساب)** ذخیره می‌شود.

ویژگی متمایز آن این است که مقدار فیلد متادیتای **executable** برابر با **true** تنظیم شده است.

این مقدار به **Runtime (محیط اجرایی)** شبکه اعلام می‌کند که این حساب شامل **کدی است که باید اجرا شود**.

---

## Stateless (بدون وضعیت)

این مهم‌ترین مفهوم در معماری برنامه‌های سولانا است.

برنامه‌ها در حساب خود **هیچ داده کاربری، متغیر یا وضعیت (State)** ذخیره نمی‌کنند.

اگر یک برنامه نیاز داشته باشد داده‌ای را ذخیره کند، باید از یک **Account (حساب)** جداگانه و خارجی استفاده کند که به آن **State Account (حساب وضعیت)** گفته می‌شود.

---

# Instructions vs Transactions (دستورها در مقابل تراکنش‌ها)

هنگام تعامل با بلاک‌چین، باید بین دو مفهوم تمایز قائل شد:

- **عمل خاصی که می‌خواهید انجام دهید**
- **وسیله‌ای که آن عمل را به شبکه ارسال می‌کند**

---

## The Instruction (ix) — دستور

یک **Instruction (دستور)** در واقع یک **فراخوانی تابع به یک Program (برنامه)** است.

این دستور مشخص می‌کند که:

- کدام برنامه اجرا شود
- چه منطقی اجرا گردد

به عبارت دیگر، **Instruction** یک **درخواست مشخص برای اجرای منطق برنامه** است.

---

## The Transaction (tx) — تراکنش

یک **Transaction (تراکنش)** در واقع **بسته‌ای از چند Instruction** است.

این همان **واحد اتمی** است که به شبکه سولانا ارسال می‌شود.

ویژگی‌ها:

- یک تراکنش می‌تواند **یک یا چند Instruction** داشته باشد.
- این دستورها می‌توانند:
  - چند بار یک برنامه را فراخوانی کنند
  - یا چند **Program (برنامه)** متفاوت را به صورت پشت‌سرهم اجرا کنند.

### Atomicity (اتمی بودن)

اگر **هر کدام از Instructionها شکست بخورد**، کل **Transaction** شکست می‌خورد.

این ویژگی باعث می‌شود عملیات **اتمی (Atomic)** باقی بماند.

---

# Structure of a Transaction (ساختار یک تراکنش)

برای درک نحوه پردازش درخواست‌ها در سولانا، ساختار یک **Transaction Object (شیء تراکنش)** را بررسی می‌کنیم.

فرض کنید کاربری به نام **Alice (آلیس)** می‌خواهد یک **Counter Program (برنامه شمارنده)** را با مقدار اولیه **0** راه‌اندازی کند.

ساختار تراکنش به صورت زیر خواهد بود:

```
tx = {
    instructions: [
        "call counter program",
        "initialize counter state"
    ],
    accounts: [
        Alice,
        Counter Account,
        System Program
    ]
}
```

### توضیح اجزا

- **instructions**
  - منطق دقیق اجرا را مشخص می‌کند

- **accounts**
  - لیست تمام حساب‌هایی که در این تراکنش درگیر هستند

در این مثال:

- **Alice** → امضاکننده و پرداخت‌کننده هزینه  
- **Counter Account** → حسابی که وضعیت شمارنده در آن ذخیره می‌شود  
- **System Program** → برنامه سیستمی مورد نیاز برای ایجاد حساب جدید  

---

# Logic Behind Account Declaration (منطق اعلام حساب‌ها)

ممکن است متوجه شده باشید که تمام حساب‌های درگیر باید **از قبل در آرایه accounts اعلام شوند**.

این موضوع یک الزام تصادفی نیست؛ بلکه **مکانیزم اصلی سرعت بالای سولانا** است.

این معماری امکان **Parallel Execution (اجرای موازی)** را فراهم می‌کند.

---

## Parallel Execution (اجرای موازی)

قبل از اجرای تراکنش، **Solana Runtime (محیط اجرایی سولانا)** که با نام **Sealevel Runtime** شناخته می‌شود، لیست حساب‌ها را بررسی می‌کند.

به مثال زیر توجه کنید:

```
Transaction A accounts:
[Alice, Counter Account]

Transaction B accounts:
[Carol, Bob]
```

از آنجا که **هیچ همپوشانی بین حساب‌ها وجود ندارد**، Runtime می‌داند که این تراکنش‌ها **به وضعیت یکسانی دسترسی ندارند**.

در نتیجه:

- این تراکنش‌ها **به صورت همزمان اجرا می‌شوند**
- نیازی به اجرای **ترتیبی (Sequential Execution)** نیست

این معماری یکی از دلایل اصلی توانایی سولانا برای پردازش **هزاران تراکنش در ثانیه** است.

---

# Step-by-Step Execution: Initializing a Counter Program
# اجرای مرحله‌به‌مرحله: راه‌اندازی یک Counter Program

برای درک بهتر تعامل بین **Program (برنامه)**، **State (وضعیت)** و **System Program (برنامه سیستمی)**، اجرای این سناریو را قدم‌به‌قدم بررسی می‌کنیم.

---

## 1. Submission (ارسال تراکنش)

آلیس (Alice) یک تراکنش ایجاد می‌کند که شامل دستور زیر است:

```
initialize counter state
```

سپس:

- تراکنش را **امضا (Sign)** می‌کند
- اجازه استفاده از **SOL** خود برای پرداخت **Fee (کارمزد)** و **Rent (اجاره ذخیره‌سازی)** را می‌دهد
- تراکنش را به **Solana Cluster (شبکه سولانا)** ارسال می‌کند

---

## 2. Program Invocation (فراخوانی برنامه)

سولانا رانتایم **Solana Runtime** تراکنش را دریافت می‌کند.

سپس:

- تشخیص می‌دهد که هدف تراکنش **Counter Program** است
- برنامه را با داده دستور اجرا می‌کند

```
ix = init counter state
```

---

## 3. Creating the State Account (ایجاد حساب وضعیت)

برنامه شمارنده **Counter Program** شروع به اجرای منطق خود می‌کند.

این برنامه باید مقدار **0** را ذخیره کند.

اما چون برنامه **Stateless (بدون وضعیت)** است، نمی‌تواند این متغیر را در حساب خود ذخیره کند.

در نتیجه باید یک حساب جدید به نام:

**Counter Account**

ایجاد کند.

اما یک محدودیت وجود دارد:

برنامه‌های معمولی **نمی‌توانند مستقیماً حساب جدید ایجاد کنند**.

در سولانا فقط **System Program** اجازه ایجاد حساب جدید را دارد.

---

## 4. Cross‑Program Invocation (فراخوانی بین‌برنامه‌ای)

در این مرحله، **Counter Program** یک:

**Cross‑Program Invocation (CPI) — فراخوانی بین برنامه‌ای**

انجام می‌دهد.

این یعنی برنامه یک **Instruction جدید** برای **System Program** ارسال می‌کند.

درخواست:

```
create account
```

در این فرآیند، **System Program** هزینه‌ای به نام **Rent (اجاره فضای ذخیره‌سازی)** دریافت می‌کند.

**Rent** مقدار **SOL** است که باید در حساب قفل شود تا هزینه ذخیره داده روی **Validator (اعتبارسنج)**‌های شبکه پرداخت شود.

در این مثال:

**Alice** که در لیست حساب‌ها به عنوان **Signer (امضاکننده)** اعلام شده، این هزینه را پرداخت می‌کند.

---

## 5. Ownership Assignment (تخصیص مالکیت)

**System Program** اکنون **Counter Account** جدید را ایجاد می‌کند.

نکته بسیار مهم:

مالک این حساب جدید را **Counter Program** قرار می‌دهد.

این کار مطابق با **مدل امنیتی سخت‌گیرانه سولانا** است:

- فقط **Owner (مالک)** یک حساب اجازه تغییر **data** آن را دارد.

با اختصاص مالکیت به **Counter Program**، این برنامه اجازه پیدا می‌کند در آینده داده‌های این حساب را تغییر دهد.

---

## 6. Writing State (نوشتن وضعیت)

اکنون که:

- حساب **Counter Account** ایجاد شده
- مالکیت آن به **Counter Program** اختصاص یافته

برنامه می‌تواند داده را در آن ذخیره کند.

در این مرحله مقدار زیر نوشته می‌شود:

```
count = 0
```

تراکنش با موفقیت تکمیل می‌شود و وضعیت جدید روی **Blockchain (بلاک‌چین)** ذخیره می‌شود.

---

# Key Takeaways (نکات کلیدی)

- **Stateless Architecture (معماری بدون وضعیت)**  
  در سولانا، **Logic (منطق برنامه)** و **Data (داده یا وضعیت)** کاملاً از یکدیگر جدا هستند. این معماری با بلاک‌چین‌های **EVM** که قراردادها وضعیت خود را نگهداری می‌کنند تفاوت اساسی دارد.

- **Rent (هزینه ذخیره‌سازی)**  
  ذخیره وضعیت در بلاک‌چین نیازمند فضای ذخیره است. این هزینه به شکل **SOL** و تحت عنوان **Rent** توسط کاربری که تراکنش را ارسال می‌کند پرداخت می‌شود.

- **Efficiency via Declaration (کارایی از طریق اعلام حساب‌ها)**  
  اعلام تمام حساب‌های درگیر در ابتدای تراکنش به Runtime اجازه می‌دهد تراکنش‌های بدون تداخل را شناسایی کرده و آن‌ها را **به صورت موازی اجرا کند**.

- **Ownership Rules (قوانین مالکیت)**  
  امنیت داده‌ها از طریق **مالکیت حساب‌ها** تضمین می‌شود. یک برنامه فقط می‌تواند داده حساب‌هایی را تغییر دهد که **مالک رسمی آن‌ها** باشد.
