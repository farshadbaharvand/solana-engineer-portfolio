# Understanding Solana Accounts (درک حساب‌های سولانا): مدل داده‌ای اصلی شبکه

اگر از **Ethereum (اتریوم)** یا توسعه سنتی **Web Development (توسعه وب)** به **Solana (سولانا)** مهاجرت می‌کنید، قبل از نوشتن حتی یک خط کد باید یک مفهوم بنیادی برنامه‌نویسی را درک کنید: **Accounts (حساب‌ها)**.

در بسیاری از معماری‌های بلاک‌چین، **Logic (منطق / قراردادهای هوشمند)** و **Data (داده / وضعیت)** معمولاً در کنار یکدیگر ذخیره می‌شوند. اما **Solana** رویکرد متفاوتی دارد.

قانون طلایی توسعه در سولانا بسیار ساده است:

**تمام داده‌ها در Solana داخل Accounts ذخیره می‌شوند.**

چه در حال مشاهده موارد زیر باشید:

- موجودی کیف پول یک کاربر
- یک **Smart Contract (قرارداد هوشمند)** مستقر شده
- متادیتای یک **NFT (توکن غیرقابل تعویض)**

در واقع در حال تعامل با یک **Solana Account (حساب سولانا)** هستید.

درک **ساختار و آناتومی این حساب‌ها** اولین قدم برای تسلط بر پروتکل سولانا است.

---

## Anatomy of a Solana Account (آناتومی یک حساب سولانا)

در سطح پایین، هر **Account** در سولانا از یک ساختار داده مشخص پیروی می‌کند. اگر به تعریف آن در کد منبع نگاه کنید، **Account** یک **Struct (ساختار)** است که شامل چند فیلد مشخص می‌شود.

ساختار آن در زبان **Rust** به شکل زیر است:

```rust
Account {
    lamports: u64,
    data: Vec<u8>,
    owner: Pubkey,
    executable: bool,
    // rent_epoch (deprecated)
}
```

اکنون هر یک از این فیلدها را بررسی می‌کنیم تا ببینیم سولانا چگونه **State (وضعیت)**، **Currency (ارز)** و **Logic (منطق)** را مدیریت می‌کند.

---

## 1. Lamports (لامپورت‌ها): مدیریت موجودی و Rent

فیلد **lamports** نشان‌دهنده **موجودی حساب** است.

همان‌طور که **Ethereum** از **Wei (وی)** برای اندازه‌گیری واحدهای کوچک **ETH** استفاده می‌کند، **Solana** نیز از **Lamports (لامپورت)** استفاده می‌کند.

### Conversion Rate (نرخ تبدیل)

```
1 SOL = 1,000,000,000 Lamports
```

---

### Concept of Rent (مفهوم Rent)

در **Solana** ذخیره‌سازی داده رایگان نیست.

وقتی یک **Account** برای ذخیره داده ایجاد می‌کنید — چه برای:

- موجودی توکن
- داده‌های یک برنامه
- یا یک **Smart Contract**

در واقع از **فضای ذخیره‌سازی فیزیکی** روی **Validator (اعتبارسنج)**های شبکه استفاده می‌کنید.

برای پرداخت هزینه این فضا، حساب باید **حداقل مقدار مشخصی Lamports** داشته باشد. این مکانیزم **Rent (اجاره فضای ذخیره‌سازی)** نامیده می‌شود.

---

### Rent‑Exempt (معاف از Rent)

اگر یک حساب مقدار کافی **SOL** برای پوشش هزینه ذخیره‌سازی داده به مدت **دو سال** داشته باشد، به آن:

**Rent‑Exempt Account (حساب معاف از اجاره)**

گفته می‌شود.

این حالت در توسعه مدرن سولانا **استاندارد رایج** محسوب می‌شود.

---

### Proportional Cost (هزینه متناسب با داده)

هرچه اندازه فیلد **data** بزرگ‌تر باشد:

- فضای ذخیره بیشتری مصرف می‌شود
- مقدار **SOL** بیشتری باید در فیلد **lamports** قفل شود

---

## 2. Data (داده): Storage vs Logic

فیلد **data** یک **Byte Array (آرایه بایت)** از نوع **Vec<u8>** است که می‌تواند هر نوع اطلاعاتی را ذخیره کند.

عملکرد این فیلد به **نوع Account** بستگی دارد.

### Storage Accounts (حساب‌های ذخیره‌سازی)

اگر حساب برای ذخیره **State (وضعیت)** استفاده شود، این فیلد شامل متغیرها خواهد بود.

مثال‌ها:

- شمارنده (Counter)
- پروفایل کاربر
- موجودی توکن

---

### Program Accounts (حساب‌های برنامه)

اگر حساب یک **Smart Contract (قرارداد هوشمند)** باشد، این فیلد شامل:

**Executable Bytecode (بایت‌کد اجرایی)**

خواهد بود.

---

### Separation of Concerns (تفکیک مسئولیت‌ها)

سولانا به‌طور سخت‌گیرانه‌ای **Code (کد)** و **State (وضعیت)** را از هم جدا می‌کند.

- **Program Accounts** → ذخیره **Logic (کد غیرقابل تغییر)**
- **Data Accounts** → ذخیره **State (داده قابل تغییر)**

مثال:

یک **Token Program** در یک حساب ذخیره می‌شود، اما اطلاعات مربوط به **موجودی توکن شما** در حساب کاملاً جداگانه‌ای نگهداری می‌شود.

---

## 3. Owner (مالک): مرز امنیتی

هر **Account** در سولانا یک **Owner (مالک)** دارد.

مالک یک **Pubkey (کلید عمومی)** است که به یک **Program (برنامه)** اشاره می‌کند.

این فیلد **مدل امنیتی بلاک‌چین** را تعیین می‌کند.

---

### Ownership Rules (قوانین مالکیت)

- فقط **Owner** می‌تواند **data** داخل حساب را تغییر دهد.
- فقط **Owner** می‌تواند از **lamports** حساب برداشت کند.

این ساختار تضمین می‌کند که:

- هیچ مهاجمی نمی‌تواند داده‌های شما را تغییر دهد
- هیچ‌کس نمی‌تواند دارایی شما را تخلیه کند

مگر اینکه **منطق برنامه مالک** چنین اجازه‌ای بدهد.

---

## 4. Executable (قابل اجرا): Program vs State

فیلد **executable** یک **Boolean Flag (پرچم منطقی)** است.

مقادیر ممکن:

- **true**

حساب شامل **Executable Bytecode** است و یک **Program (قرارداد هوشمند)** محسوب می‌شود.

- **false**

حساب فقط شامل **Data (داده)** است.

# Practical Examples of Solana Accounts (نمونه‌های عملی حساب در سولانا)

## Example 1: Standard User Wallet (کیف پول استاندارد کاربر)

```rust
Account {
    lamports: 1,000,000,000,
    data: [],
    owner: System Program,
    executable: false
}
```

## Example 2: Deploying a Smart Contract (استقرار قرارداد هوشمند)

```rust
Account {
    lamports: 100,000,
    data: [0, 1, 255, ...],
    owner: BPF Loader,
    executable: true
}
```

## Example 3: Storing Program State (ذخیره وضعیت برنامه)

```rust
Account {
    lamports: 1,000,
    data: [0, 0, 0, 1],
    owner: Counter Program ID,
    executable: false
}
```

---

# Summary of Account Hierarchy (خلاصه سلسله‌مراتب حساب‌ها)

- **System Program** → مدیریت **User Wallets (کیف پول کاربران)**
- **BPF Loader** → مدیریت **Custom Programs (برنامه‌های سفارشی)**
- **Custom Programs** → مدیریت **Data Accounts (حساب‌های داده)**

درک این ساختار نشان می‌دهد چگونه **Solana** می‌تواند همزمان **Throughput بالا** و **امنیت قوی** برای مدیریت داده‌ها و دارایی‌ها فراهم کند.
