# Program Derived Addresses (PDAs) در سولانا

در اکوسیستم **Solana (سولانا)**، درک **مدیریت حساب‌ها (Account Management)** برای ساخت **Decentralized Applications - dApps (اپلیکیشن‌های غیرمتمرکز)** امن و کاربردی ضروری است.  
در حالی که بسیاری از توسعه‌دهندگان با **Standard Accounts (حساب‌های استاندارد)** آشنا هستند، مفهوم **Program Derived Addresses - PDAs (آدرس‌های مشتق‌شده از برنامه)** یک مفهوم میانی اما بسیار حیاتی است که توانایی کامل مدل برنامه‌نویسی سولانا را در اختیار شما قرار می‌دهد.

برای درک PDA ابتدا باید یک **Solana Account (حساب سولانا)** معمولی را بررسی کنیم.

در حالت معمول، یک حساب توسط یک **Public/Private Key Pair (جفت کلید عمومی/خصوصی)** تعریف می‌شود:

کلید خصوصی **Private Key** برای تولید **Public Key (کلید عمومی)** استفاده می‌شود.


کلید خصوصی برای **Sign Transactions (امضای تراکنش‌ها)** مورد نیاز است.



اما **PDA** این قاعده را می‌شکند.

یک **PDA** در واقع یک **Public Key (کلید عمومی)** است که **هیچ Private Key (کلید خصوصی)** مرتبطی ندارد.  
به همین دلیل هیچ کاربر خارجی (مثل صاحب کیف پول) نمی‌تواند برای آن **Signature (امضا)** ایجاد کند.

در عوض، شبکه سولانا اجازه می‌دهد **Program (برنامه)**‌ای که این آدرس را ایجاد کرده است، به صورت **Programmatic Signing (امضای برنامه‌ای)** برای آن امضا ایجاد کند.

---

# نحوه ایجاد PDA

برخلاف **Wallet Addresses (آدرس‌های کیف پول)** معمولی، **PDAs** به صورت تصادفی ساخته نمی‌شوند.  
آن‌ها به صورت **Deterministic (قطعی)** تولید می‌شوند.

یعنی اگر دقیقاً همان ورودی‌ها را دوباره بدهید، همیشه **همان آدرس** تولید خواهد شد.

برای ساخت یک **PDA** سه ورودی لازم است:

- **Program ID (شناسه برنامه)**  

  آدرس برنامه‌ای که مالک و کنترل‌کننده **PDA** خواهد بود.

- **Seeds (سیدها)**  

  مجموعه‌ای از ورودی‌های تعریف‌شده توسط توسعه‌دهنده. این‌ها می‌توانند:
  - رشته‌ها
  - اعداد
  - کلیدهای عمومی دیگر (مثلاً آدرس کیف پول کاربر)

- **Bump (بامپ)**  

  یک عدد صحیح بین **0 تا 255** که برای خارج کردن آدرس از **Cryptographic Curve (منحنی رمزنگاری)** استفاده می‌شود.

فرمول مفهومی یک PDA به شکل زیر است:

```
PDA = hash(Program ID + Seeds + Bump)
```

---

# چرا PDAs مهم هستند؟

 پی ای دی ها **PDAs** ستون فقرات **Secure State Management (مدیریت امن وضعیت)** در سولانا هستند.  
آن‌ها سه مزیت اصلی دارند که مشکلات معماری پیچیده را حل می‌کنند.

---

## 1. امنیت و مالکیت برنامه‌ای

از آنجا که **PDA** هیچ **Private Key (کلید خصوصی)** ندارد، هیچ کاربر انسانی نمی‌تواند برای آن تراکنش امضا کند.

در عوض:

- **Solana Runtime (زمان اجرای سولانا)**

تشخیص می‌دهد که این PDA متعلق به کدام **Program ID (شناسه برنامه)** است.
- فقط همان برنامه اجازه دارد:
  - **Modify Account Data (تغییر داده حساب)**
  - **Transfer Funds (انتقال دارایی‌ها)**

این موضوع باعث می‌شود:

- نیازی به مدیریت کلیدهای خصوصی برای **Smart Contracts (قراردادهای هوشمند)** نباشد.
- محیطی **Trustless (بدون نیاز به اعتماد)** ایجاد شود که قوانین حساب‌ها کاملاً توسط کد اجرا می‌شوند.

---

## 2. ساختار داده شبیه Hashmap

 پی دی ای ها **PDAs** به توسعه‌دهندگان اجازه می‌دهند نگاشت های **Mapping** قطعی بین کاربران و داده‌های **On-Chain (روی بلاکچین)** ایجاد کنند.

این ساختار شبیه **Key-Value Store (ذخیره‌سازی کلید-مقدار)** یا **Hashmap (هش‌مپ)** عمل می‌کند.

- **Key (کلید)**  

  از ترکیب **User Public Key (کلید عمومی کاربر)** و **Program ID (شناسه برنامه)** ساخته می‌شود.

- **Value (مقدار)**  

  داده‌ای است که داخل PDA ذخیره می‌شود.

برای مثال:

یک توسعه‌دهنده می‌تواند **User Public Key (کلید عمومی کاربر)** را مستقیماً به یک حساب مثل موارد زیر نگاشت کند:

- **Locked State Account (حساب وضعیت قفل)**
- **Authorization Status Account (حساب وضعیت مجوز)**

بدون اینکه نیاز به **Centralized Database (پایگاه داده متمرکز)** برای نگهداری محل این داده‌ها باشد.

---

## 3. آدرس‌دهی قطعی

از آنجا که آدرس PDA از **Seeds (سیدها)** محاسبه می‌شود، یک **Client Application (اپلیکیشن کلاینت)** مانند **Web Frontend (فرانت‌اند وب)** می‌تواند قبل از هر تعامل با بلاکچین دقیقاً محاسبه کند که داده در کجا ذخیره خواهد شد.

مزایا:

- **UI Responsiveness (واکنش‌پذیری رابط کاربری)** بهتر
- **Transaction Construction (ساخت تراکنش)** ساده‌تر

---

# مثال واقعی: Lock Program

برای درک بهتر، یک سناریوی واقعی را در نظر بگیریم.

فرض کنید کاربری به نام **Alice** می‌خواهد **1 SOL** را به مدت **یک سال** در یک **Lock Program (برنامه قفل)** قفل کند.

---

## ورودی‌ها

- **Program ID**  

  آدرس **Lock Program**

- **Seeds**

  - **Alice Public Key**

  - یک **User Specified Random Number (عدد تصادفی انتخاب‌شده توسط کاربر)**

وجود این عدد تصادفی باعث می‌شود Alice بتواند **چند حساب قفل مختلف** ایجاد کند.

- **Bump**  

  یک عدد استاندارد مانند **255**

---

## فرآیند

### 1. Derivation (مشتق‌سازی)

 برنامه قفل **Lock Program** یا **Client** با استفاده از ورودی‌ها **PDA** را محاسبه می‌کند.

این آدرس تبدیل به **Lock Account (حساب قفل)** آلیس می‌شود.

### 2. Funding (تأمین سرمایه)

آلیس مقدار **1 SOL** را به این **PDA** ارسال می‌کند.

### 3. State Management (مدیریت وضعیت)

برنامه حساب را مقداردهی اولیه می‌کند و داده‌هایی مثل موارد زیر را ذخیره می‌کند:

 - زمان انقضای قفل  **Lock Expiry Timestamp**

---

## نتیجه

اگرچه این **PDA** از نظر رمزنگاری به آلیس مرتبط است (چون کلید او در Seeds بوده)، اما **آلیس نمی‌تواند مستقیماً پول را برداشت کند**.

فقط **Lock Program** می‌تواند **1 SOL** را از این حساب منتقل کند.

کد برنامه:

- زمان فعلی را بررسی می‌کند
- آن را با **Lock Expiry Timestamp** مقایسه می‌کند
- فقط در صورت رسیدن زمان، **Withdrawal Transaction (تراکنش برداشت)** را امضا می‌کند.

---

# بررسی فنی: Bump و منحنی Ed25519

فنی‌ترین بخش **PDA** مفهوم **Bump (بامپ)** و دلیل نیاز به آن است.

سولانا از **Ed25519 Digital Signature Scheme (الگوریتم امضای دیجیتال Ed25519)** استفاده می‌کند.

در این سیستم رمزنگاری:

- **Standard Public Keys (کلیدهای عمومی معمولی)** نقاطی هستند که **روی منحنی بیضوی Ed25519** قرار دارند.
- **PDAs** باید نقاطی باشند که **خارج از این منحنی** قرار دارند.

چرا؟

اگر یک آدرس:

- **روی منحنی (On Curve)** باشد → یک **Private Key (کلید خصوصی)** دارد.
- **خارج از منحنی (Off Curve)** باشد → تولید کلید خصوصی **از نظر ریاضی غیرممکن** است.

---

## نقش Bump

وقتی سیستم می‌خواهد یک **PDA** با استفاده از **Program ID** و **Seeds** بسازد:

یک **Hash Function (تابع هش)** اجرا می‌شود.

اما تقریباً **۵۰٪ احتمال** دارد که نتیجه **روی منحنی** قرار بگیرد.

که در این صورت:

- یک **Public Key معتبر**
- اما **PDA نامعتبر**

خواهد بود.

---

### مکانیزم Bump

**Bump** مانند یک **Nonce (عدد یک‌بارمصرف)** عمل می‌کند.

فرآیند:

1. سیستم با **bump = 255** شروع می‌کند.
2. ورودی‌ها هش می‌شوند.
3. بررسی می‌شود:

```
Is this point on the curve?
```

- اگر **YES (On Curve)**  
  برای PDA نامعتبر است.  
  مقدار bump یک واحد کاهش می‌یابد.

```
255 → 254 → 253 ...
```

- اگر **NO (Off Curve)**  
  یک **PDA معتبر** تولید شده است.

این مکانیزم تضمین می‌کند که:

- آدرس **کلید خصوصی ندارد**
- فقط به عنوان **Storage Account (حساب ذخیره‌سازی)** برای برنامه عمل می‌کند.

---

# جمع‌بندی و نکات کلیدی

تسلط بر **Program Derived Addresses (PDAs)** برای نوشتن برنامه‌های پیشرفته در سولانا ضروری است.

مفاهیم اصلی:

- **No Private Keys (بدون کلید خصوصی)**  
  PDAها کلید خصوصی ندارند و فقط برنامه مالک می‌تواند از طریق **Solana Runtime** برای آن‌ها امضا کند.

- **Deterministic Mapping (نگاشت قطعی)**  
  مانند یک **On-Chain Key-Value Store (ذخیره‌سازی کلید-مقدار روی بلاکچین)** عمل می‌کنند.

- **Uniqueness (یگانگی)**  
  با افزودن **Seeds (سیدهای منحصر‌به‌فرد)** مثل ID یا عدد تصادفی، یک کاربر می‌تواند چندین PDA از یک برنامه داشته باشد.

- **Security (امنیت)**  
  منطق برنامه کنترل دارایی‌ها را بر عهده دارد؛ مانند مثال **Lock Program** که در آن دارایی‌ها به صورت برنامه‌ای قفل می‌شوند.
