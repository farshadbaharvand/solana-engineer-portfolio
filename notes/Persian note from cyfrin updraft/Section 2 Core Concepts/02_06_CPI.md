# Cross Program Invocation (CPI) در سولانا

در قلب اکوسیستم **Solana (سولانا)** مفهومی به نام **Composability (ترکیب‌پذیری)** قرار دارد.  
این مفهوم به **Decentralized Applications - dApps (اپلیکیشن‌های غیرمتمرکز)** اجازه می‌دهد با یکدیگر تعامل داشته باشند، روی هم ساخته شوند و مانند مجموعه‌ای از **Building Blocks (اجزای سازنده)** به هم متصل عمل کنند.

مکانیزم فنی که این قابلیت را فراهم می‌کند **Cross Program Invocation - CPI (فراخوانی میان‌برنامه‌ای)** نام دارد.

یک **CPI** زمانی رخ می‌دهد که یک **Program (برنامه)** در حال اجرا، مستقیماً 
دستورهای یک برنامه دیگر را فراخوانی کند.

---

## یک مثال ساده برای درک CPI

برای درک بهتر، یک مثال از دنیای **Web2 (وب سنتی)** را در نظر بگیرید.

فرض کنید یک **Server-side API Endpoint (اندپوینت API سمت سرور)** در حین اجرا، یک **Internal API Endpoint (اندپوینت داخلی)** دیگر را فراخوانی می‌کند تا:

- داده‌ای دریافت کند
- یا یک عملیات خاص انجام دهد

در **Solana** نیز **CPI** دقیقاً همین رفتار را دارد.

برای مثال، اگر تاکنون یک **Counter Program (برنامه شمارنده)** ساخته باشید که یک حساب را **Initialize (مقداردهی اولیه)** می‌کند، احتمالاً برنامه شما **System Program (برنامه سیستمی)** را فراخوانی کرده است.

آن تعامل در واقع یک **Cross Program Invocation (فراخوانی میان‌برنامه‌ای)** بوده است.

---

# Privilege Extension (گسترش دسترسی)

در تراکنش‌های بلاکچین، **Security (امنیت)** و **Authority (اختیار)** اهمیت بسیار زیادی دارند.

یکی از مهم‌ترین قوانین در **CPI** مفهوم **Privilege Extension (گسترش سطح دسترسی)** است.

این مفهوم مشخص می‌کند چگونه موارد زیر از یک برنامه به برنامه دیگر منتقل می‌شوند:

- **Signing Authority (اختیار امضا)**
- **Access Rights (حقوق دسترسی)** مانند قابلیت نوشتن

در اینجا دو نقش وجود دارد:

- **Caller (برنامه فراخواننده)**
- **Callee (برنامه فراخوانده‌شده)**

---

## جریان یک تراکنش

سناریویی را در نظر بگیرید که شامل یک کاربر و دو برنامه است:

- **Program A**
- **Program B**

---

### نقش کاربر

یک کاربر (حساب **Wallet Account (کیف پول)**) یک **Transaction (تراکنش)** را برای تعامل با **Program A** امضا می‌کند.

در این مرحله کاربر مشخص می‌کند که کدام حساب‌ها:

- **Signer (امضاکننده)** هستند  
- **Writable (قابل نوشتن)** هستند

---

### برنامه فراخواننده (Program A)

برنامه الف **Program A** شروع به اجرا می‌کند.

این برنامه در حال حاضر دارای مجوزهایی است که کاربر در تراکنش تعیین کرده است.

---

### اجرای CPI (فراخوانی Program B)

در حین اجرای منطق خود، **Program A** تصمیم می‌گیرد یک **Instruction (دستور)** از **Program B** را فراخوانی کند.

---

## انتقال دسترسی‌ها

زمانی که **Program A** یک **CPI** انجام می‌دهد، می‌تواند حساب‌هایی را که از کاربر دریافت کرده است به **Program B** ارسال کند.

نکته مهم:

**مجوزهای اصلی این حساب‌ها حفظ می‌شوند.**

- اگر یک حساب برای **Program A** به عنوان **Signer (امضاکننده)** تعریف شده باشد، برای **Program B** نیز **Signer** باقی می‌ماند.

- اگر یک حساب برای **Program A** به عنوان **Writable (قابل نوشتن)** تعریف شده باشد، برای **Program B** نیز **Writable** خواهد بود.

بنابراین **Program B** می‌تواند عملیات‌های دارای دسترسی بالا انجام دهد زیرا مجوزها در مسیر زیر منتقل شده‌اند:

```
User
   ↓
Program A
   ↓
Program B
```

---

# امضای PDA

در بسیاری از موارد، برنامه‌ها باید به جای کاربران برای خودشان امضا ایجاد کنند.

این کار با استفاده از **Program Derived Addresses - PDA (آدرس‌های مشتق‌شده از برنامه)** انجام می‌شود.

اگر **Program A** از طرف **PDA** خود امضا کند، این **Signer Privileges (مجوز امضا)** نیز به برنامه فراخوانده‌شده منتقل می‌شود.

این قابلیت اجازه می‌دهد:

- قراردادها به صورت **Autonomous (خودکار)** با هم تعامل داشته باشند
- کاربر مجبور نباشد برای هر برنامه در زنجیره یک **Transaction Signature (امضای تراکنش)** جداگانه انجام دهد

در نتیجه می‌توان **Smart Contract Interactions (تعامل قراردادهای هوشمند)** پیچیده ایجاد کرد.

---

# Maximum CPI Depth (حداکثر عمق CPI)

اگرچه **CPI** امکان زنجیره‌ای از فراخوانی‌ها را فراهم می‌کند، اما پروتکل **Solana** محدودیتی برای تعداد فراخوانی‌های تو در تو در یک تراکنش اعمال کرده است.

به این محدودیت **Maximum CPI Depth (حداکثر عمق CPI)** گفته می‌شود.

این محدودیت برای جلوگیری از:

- **Infinite Loops (حلقه‌های بی‌نهایت)**
- مصرف بیش از حد **Computational Resources (منابع محاسباتی)**

اعمال شده است.

---

## محدودیت Stack

در حال حاضر حداکثر عمق **Instruction Stack (پشته دستورها)** برابر با **4** است.

---

### مثال از زنجیره فراخوانی

فرض کنید یک زنجیره از برنامه‌ها وجود دارد:

- **Program A** → فراخوانی **Program B** (عمق 1)

- **Program B** → فراخوانی **Program C** (عمق 2)

- **Program C** → فراخوانی **Program D** (عمق 3)

- **Program D** → فراخوانی **Program E** (عمق 4)

در این سناریو:

**Program E** به آخرین سطح پشته رسیده است.

این برنامه می‌تواند منطق خود را اجرا کند، اما **نمی‌تواند یک CPI دیگر به برنامه‌ای مانند Program F انجام دهد.**

---

# محدودیت فنی در کد سولانا

در **Solana Source Code (کد منبع سولانا)** و مستندات آن، این محدودیت با ثابت زیر تعریف شده است:

```
MAX_INSTRUCTION_STACK_DEPTH
```

توسعه‌دهندگان باید توجه داشته باشند که مقدار **4** محدودیت فعلی پروتکل است.

اما با پیشرفت شبکه **Solana** ممکن است این مقدار در **Protocol Updates (به‌روزرسانی‌های پروتکل)** آینده تغییر کند.

---

# جمع‌بندی

فراخوانی میان‌برنامه‌ای **Cross Program Invocation (CPI)** مانند چسبی است که کل اکوسیستم **Solana** را به هم متصل می‌کند.

درک صحیح **Privilege Extension (گسترش دسترسی)** به توسعه‌دهندگان کمک می‌کند تا برنامه‌هایشان بتوانند به شکل **Secure (امن)** و **Trustless (بدون نیاز به اعتماد)** از طرف کاربران در چندین قرارداد عمل کنند.

همچنین آگاهی از **CPI Depth (عمق CPI)** به توسعه‌دهندگان کمک می‌کند معماری برنامه خود را:

- **Efficient (کارآمد)**
- سازگار با محدودیت‌های **Computational Limits (محدودیت‌های محاسباتی پروتکل)**

طراحی کنند.
